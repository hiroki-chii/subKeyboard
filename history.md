# SubKeyboardV2 開発履歴

## 現状の課題
- なし（100%の動作安定性を追求）。

## 解決済み
- **リークとチャタリングの完全解決**: グローバルキーボードフック（`WH_KEYBOARD_LL`）による「全キー一旦停止・判定後再注入」方式を採用。システム全体でリークを完全に封じ込めつつ、`LLKHF_INJECTED` フラグにより無限ループ（チャタリング）を回避。
- **WebView内入力の復旧**: `RIDEV_NOLEGACY` 廃止と全ウィンドウサブクラス化の組み合わせ。
- **永続化と命名規則**: 実装・修正済み。

## 開発記録

### 2026-04-21 11:32
- **解決策 (真の究極版)**:
    - **グローバルフック (WH_KEYBOARD_LL)**: すべての物理入力を一旦システム全体で swallow（飲み込み）する。
    - **非同期判定**: `PostMessage` を使ってメインウィンドウに判定を委ね、Raw Input の情報と照らし合わせる。
    - **スレッドセーフ対策**: `HHOOK` や `HWND` が生ポインタを内包しており `Sync` でないため、`Sendable` ラッパーを導入してグローバルな `Mutex` に安全に格納できるように修正。
    - **結果**: 他のアプリ（メモ帳等）へのリークもゼロになり、チャタリング（無限ループ）も解消。

### 2026-04-21 11:22
- **不具合**: `RIDEV_NOLEGACY` 方式では、`SendInput` の再注入が再度 Raw Input を誘発し、無限ループ（チャタリング）が発生した。また、`NOLEGACY` は登録ウィンドウに対してのみ有効で、システム全体のリーク防止には不十分だった。

### 2026-04-21 11:06
- **不具合**: マクロは動くが、依然としてリーク（実行キーの入力）が発生する。
- **原因**: 
    - `PeekMessage` は「既にキューにあるもの」を消すだけなので、`WM_INPUT` と `WM_KEYDOWN` が極めて近いタイミングで処理される際に、タイミング次第で見逃してしまう。
- **対策**: 
    - 「能動的トラップ方式」を導入。`WM_INPUT` を受け取った直後の3つのキーメッセージ（KEYDOWN, CHAR, KEYUP等）を強制的に破棄するカウンタを設けた。
    - これにより、トレードオフなしで「マクロ実行」と「リーク抑制」を実現。

### 2026-04-21 11:02
- **不具合**: 文字入力不能・マクロ不動。
- **原因**: `RIDEV_NOLEGACY` の副作用。
- **対策**: `NOLEGACY` 廃止、全ウィンドウ再帰サブクラス化。
